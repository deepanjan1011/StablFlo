# Dev Panel Spec

**Goal:** A floating developer panel for testing app states during demo — skip onboarding, inject claims, adjust coverage, and reset — without touching the backend.

**Approach:** Pure frontend state injection. The panel receives callback props from `page.tsx` and calls them to update React state directly. A `devMode` flag in `page.tsx` suppresses the polling `useEffect` when mock state is active. Only renders in development builds.

---

## Design Language

Matches the existing fintech dark aesthetic:
- Background: `#0f0f0f`, border-top: `1px solid #222`, top corners: `20px` radius
- Handle pill: `32px × 3px`, `#2a2a2a`
- Section cards: `#111`, border `#1a1a1a`, `10px` radius
- Floating button: `36px` circle, `#1a1a1a`, border `#333`, `⚙` icon
- Button fixed position: `bottom: 80px, right: 16px` (above bottom nav)
- Panel z-index above everything (`z-50`), same slide-up pattern as ProfileDrawer

## Component: `DevPanel`

**File:** `frontend/src/lib/DevPanel.tsx`

### Inline types (defined in this file)

```ts
type Zone = { id: number; name: string; city: string; base_premium: number; status: string };
type Policy = { id: number; max_coverage: number; premium_paid: number; start_date: string; end_date: string; is_active: boolean };
type Claim = { id: number; trigger_type: string; amount: number; timestamp: string; status: string };
```

### Props

```ts
interface DevPanelProps {
  zones: Zone[];                              // already-fetched zones from page.tsx
  activePolicy: Policy | null;                // current policy, used to seed coverage local state
  onSkipOnboarding: (zone: Zone) => void;     // jump to dashboard with mock data
  onInjectClaim: (claim: Claim) => void;      // append a claim to live claims array
  onSetCoverage: (amount: number) => void;    // update activePolicy.max_coverage
  onReset: () => void;                        // reset to onboarding (step 0)
}
```

### Local state

```ts
const [isPanelOpen, setIsPanelOpen] = useState(false);
const [coverage, setCoverage] = useState(activePolicy?.max_coverage ?? 32000);
```

`coverage` initialises from `activePolicy.max_coverage` so the `±` control reflects the current live value.

### Behaviour

- Floating `⚙` button fixed bottom-right, always on top
- Click → `isPanelOpen` toggled
- Panel open: backdrop `rgba(0,0,0,0.5)`, tap backdrop → closes
- Body scroll lock when panel open (same `useEffect` pattern as ProfileDrawer)

### Controls

**Skip Onboarding**
- Single "GO" button
- Calls `onSkipOnboarding(zones[0])` — uses the first available zone
- Mock data constructed inside DevPanel and passed via the callback mechanism (see `handleSkipOnboarding` below)

**Inject Claims**
- Three pill buttons: 🌧️ Rain ₹800 / ☀️ Heat ₹600 / 🏭 AQI ₹700
- Each calls `onInjectClaim(claim)` with:
  - `id`: `Date.now()`
  - `trigger_type`: `"rain"` / `"heat"` / `"aqi"`
  - `amount`: 800 / 600 / 700
  - `timestamp`: `new Date().toISOString()`
  - `status`: `"paid"`

**Coverage ±**
- `−` decreases `coverage` by 5000, min 5000; `+` increases by 5000, max 100000
- On every change calls `onSetCoverage(newCoverage)` immediately

**Reset**
- Red-tinted card, "RESET" button
- Calls `onReset()`

## Changes to `page.tsx`

### Import — use `next/dynamic` to exclude from production bundle

```tsx
import dynamic from "next/dynamic";
const DevPanel = dynamic(
  () => import("@/lib/DevPanel").then(m => ({ default: m.DevPanel })),
  { ssr: false }
);
```

This ensures the DevPanel module is never bundled in production — it is only loaded when the component actually renders (which the `NODE_ENV` guard prevents in prod).

### New state

```tsx
const [devMode, setDevMode] = useState(false);
```

### Polling `useEffect` — add `devMode` guard

The existing effect:
```tsx
useEffect(() => {
  if (step === 1 && riderId) {
    // ... loadDashboard + setInterval
  }
}, [step, riderId]);
```

Changes to:
```tsx
useEffect(() => {
  if (step === 1 && riderId && !devMode) {
    // ... loadDashboard + setInterval (unchanged)
  }
}, [step, riderId, devMode]);
```

This prevents the polling loop from firing and overwriting mock state when the dev panel has taken control.

### Callback implementations

```tsx
function handleSkipOnboarding(zone: Zone) {
  setDevMode(true);
  setPhone("9876543210");
  setRiderId(1);
  setSelectedZone(zone);
  setActivePolicy({
    id: 1,
    max_coverage: 32000,
    premium_paid: 799,
    start_date: new Date(Date.now() - 7 * 86400000).toISOString(),
    end_date: new Date(Date.now() + 7 * 86400000).toISOString(),
    is_active: true,
  });
  setClaims([]);
  setAllPolicies([]);
  setStep(1);
}

function handleInjectClaim(claim: Claim) {
  setClaims(prev => [claim, ...prev]);
}

function handleSetCoverage(amount: number) {
  setActivePolicy(prev => prev ? { ...prev, max_coverage: amount } : prev);
}

function handleReset() {
  setDevMode(false);
  setStep(0);
  setRiderId(null);
  setActivePolicy(null);
  setAllPolicies([]);
  setClaims([]);
  setSelectedZone(null);
  setPhone("");
  setIsProfileOpen(false);
}
```

### Render

```tsx
{process.env.NODE_ENV === "development" && (
  <DevPanel
    zones={zones}
    activePolicy={activePolicy}
    onSkipOnboarding={handleSkipOnboarding}
    onInjectClaim={handleInjectClaim}
    onSetCoverage={handleSetCoverage}
    onReset={handleReset}
  />
)}
```

Placed at the root level of the dashboard return, alongside `<ProfileDrawer>`.

## Out of Scope

- Zone switcher (first zone always used for Skip Onboarding)
- No-policy state simulation (reset + don't skip covers this)
- Persisting dev state across page refresh
- Any backend calls from the panel
